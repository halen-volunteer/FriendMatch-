# FriendMatch 前后端参数对比文档

> 基准文档：`VueFormatCode/ReadMD/后端接口全量版.md`
> 目的：核对前端实际传参与后端接口定义是否一致，优先覆盖当前已实现且联调频繁的核心接口。
> 更新时间：2026-04-14

---

## 1. 对比说明

本文档按“接口 -> 后端要求 -> 前端当前传参/收参 -> 是否一致 -> 结论/处理建议”的方式整理。

判定标准：
- 一致：前端字段名、字段含义、枚举值与后端一致
- 基本一致：主字段一致，但展示字段或兼容字段有差异
- 不一致：字段名、枚举值或结构不匹配，需要调整

## 2. 已核对接口

### 2.1 `POST /api/appeal/submit`

**后端要求**
- `relatedReportId`
- `relatedReportType`
- `appellantType`
- `appealReason`
- `appealEvidence`
- `relatedPunishId`

**前端当前情况**
- `src/views/report/AppealView.vue`
  - 传：`relatedReportId`、`relatedReportType`、`appellantType`、`appealReason`
- `src/views/user/AppealView.vue`
  - 传：`relatedPunishId`、`appealReason`、`appealEvidence`、`appellantType`
  - 同时显式传了 `relatedReportId: undefined`、`relatedReportType: undefined`

**是否一致**
- 基本一致

**结论**
- 该接口后端支持多种申诉来源，前端两个页面分别覆盖了“基于举报发起申诉”和“基于处罚记录发起申诉”两种场景。
- 当前无需强制修改。

---

### 2.2 `GET /api/appeal/my`

**后端返回核心字段**
- `id`
- `appellantId`
- `relatedReportId`
- `relatedReportType`
- `relatedPunishId`
- `appealRound`
- `appealReason`
- `appealEvidence`
- `appealStatus`
- `adminId`
- `adminReply`
- `processTime`
- `createTime`

**前端当前情况**
- `src/views/report/AppealView.vue`
  - 使用：`id`、`relatedReportId`、`appealReason`、`appealStatus`、`adminReply`
- `src/views/user/AppealView.vue`
  - 使用：`relatedPunishId`、`appealReason`、`createTime`、`appealStatus`、`adminReply`

**是否一致**
- 已一致

**结论**
- 用户侧两个申诉页面的状态与处理结果展示字段，现已统一到后端真实字段 `appealStatus/adminReply`。

---

### 2.3 `GET /api/appeal/pending`

**后端要求/返回**
- 入参：`page`、`pageSize`
- 返回：分页列表
- 当前规则：仅返回“当前管理员自己被分配到”的待处理申诉

**前端当前情况**
- `src/api/admin.js`
  - 传参：`page`、`pageSize`
- `src/views/admin/AdminAppealsView.vue`
  - 使用分页参数拉取列表

**是否一致**
- 一致

**结论**
- 该接口前端无需额外传 `adminId`。
- 当前过滤逻辑完全由后端负责。

---

### 2.4 `POST /api/appeal/handle`

**后端要求**
- `appealId`
- `appealStatus`：`1-申诉成立，2-驳回`
- `adminReply`

**前端修改前**
- `AdminAppealsView.vue` 原传参：
  - `appealId`
  - `handleStatus`
  - `handleRemark`
- 且状态值使用了：
  - `2`
  - `3`

**前端修改后**
- 现在已改为：
  - `appealId`
  - `appealStatus`
  - `adminReply`
- 状态值已改为：
  - `1 = 通过`
  - `2 = 驳回`

**是否一致**
- 已一致

**结论**
- 该接口已完成前后端对齐。

## 4. 仍建议继续核对的接口

   - 已核对：前端当前用 query 参数提交 `email/emailCode/newPassword`
   - 与后端 `@RequestParam` 接收方式一致，建议继续做真实重置联调验证

4. 团队成员管理相关接口
   - 已核对：`removeMember/updateMemberRole/transferLeader` 主参数与后端 DTO 一致
   - 建议继续做真实权限联调验证（队长/管理员/普通成员）

5. `POST /api/chat/private/send` / `POST /api/chat/team/send`
   - 已核对：文本/图片/文件主字段 `msgType/msgContent/fileUrl/fileName/fileSize` 与后端 DTO 一致
   - 若后续启用表情包/`@消息` 完整交互，建议再补测 `emojiId/atUsers`

   - 当前前端传 `conversationId/msgIds`，与后端 `MsgReadDTO` 一致
   - 建议继续做已读回执真实联调验证

7. 登录、注册、重置密码、团队成员管理、聊天多消息类型
   - 建议继续做一次全量页面联调验证
