# 前端错误码与异常处理说明

## 一、统一原则

前端不要强依赖复杂错误码，先统一按以下顺序处理：

1. 先看 HTTP 状态
2. 再看 `success`
3. 最后展示 `message`

---

## 二、推荐前端处理规则

### 1. 未登录 / 登录失效

表现：
- HTTP 401
- 或业务 message 包含“未登录”“登录失效”“token 无效”

前端处理：
- 清除本地 token
- 断开 WebSocket
- 跳转登录页

### 2. 无权限

表现：
- HTTP 403
- 或业务 message 包含“无权限操作”

前端处理：
- toast 提示
- 不跳页
- 对相关按钮做禁用或隐藏

### 3. 参数错误

表现：
- `success=false`
- message 类似“参数不合法”“关键词不能为空”等

前端处理：
- 表单页做字段级提示
- 非表单页直接 toast

### 4. 业务失败

常见场景：
- 被拉黑
- 被禁言
- 被封号
- 团队不存在
- 消息已无法编辑/撤回
- 重复申请 / 重复收藏 / 重复置顶

前端处理：
- 直接展示后端 `message`
- 不自行推断额外文案

### 5. 系统错误

表现：
- HTTP 500
- 网络超时
- 网关错误

前端处理：
- 展示“系统繁忙，请稍后重试”
- 提供重试按钮

---

## 三、模块级异常提示建议

### 认证模块
- 验证码错误
- 邮箱已注册
- 验证码发送过于频繁
- 密码格式不合法

### 用户模块
- 用户不存在
- 无权查看
- 昵称包含敏感词

### 团队模块
- 团队不存在
- 无权限操作
- 已在团队中
- 申请重复

### 聊天模块
- 对方不允许私聊
- 你已被全局禁言
- 你已被团队禁言
- 消息发送失败

### 消息增强
- 仅可编辑自己的消息
- 超过编辑时间窗口
- 超过撤回时间窗口
- 该消息已被处理

### 治理安全
- 申诉次数已达上限
- 举报不存在
- 处罚记录不存在

---

## 四、前端封装建议

建议统一封装：

```ts
function handleBizError(message?: string) {
  window.$message?.error(message || '操作失败')
}
```

并在请求响应拦截器中统一处理。
